Changes 2026-05-07: c910 cmark fixes (X-cascade + parallel-dispatch stall)

Two independent bugs found and fixed for c910 cmark. With both fixes,
the test now runs to sim 1,100,000+ in parallel mode with 51,680
sustained retire events and 35.8B instructions executed, zero failures.
Pre-session: deterministic TEST FAILED at sim 49,445.

================================================================
Bug 1: X-cascade canary at sim 49,445 (testbench failure sentinel)
================================================================

Symptom:
  c910 cmark hit tb.v's failure-sentinel value 0x2382348720 on
  rbus_pipe0_wb_data at sim 49,445 with TEST FAILED. Trap was an
  illegal-instruction exception (mcause=0x2) at PC 0x485A — the CPU
  had gone off-rails earlier and was executing data as instructions.

Root cause:
  XEZIM_INIT_ZERO=1 was supposed to coerce all-X initial values to
  zero (matches `+vcs+initreg=zero`-style behavior in commercial
  simulators). The implementation only iterated `module.signals`
  (top-level reg/wire init values) but ignored `module.arrays`
  / `module.arrays_2d` / `module.arrays_nd`.

  c910 stores the PRF (96 entries x 64 bits), AIQ entries, ROB,
  and rename tables as `module.arrays`. Array elements were
  initialized via Value::new(w) which is all-X (value.rs:88).
  PRF entries that hadn't been written yet returned X through
  the forwarding mux (`fwd_dp_rf_pipe1_src1_data`), cascading
  through IDU -> IU -> ALU -> rbus -> CPU went off-rails ->
  illegal-instruction trap -> testbench sentinel hit.

  First X transition (with prior workarounds): t=42255 on
  `fwd_dp_rf_pipe1_src1_data`, originating in the case-mux that
  reads from PRF entries. PRF entries: array elements, never
  zeroed by INIT_ZERO. Tested falsified hypotheses:
   - INIT_ZERO=1 alone: still failed at sim 49,445 (because
     arrays weren't zeroed)
   - 1'bx -> 1'b0 source patch on c910 RTL (656 occurrences):
     still failed identically (literals weren't the source)
   - Combined patch + INIT_ZERO: failed identically (X moved
     from t=0 to t=42255 but pipeline still poisoned)

Fix:
  simulator.rs ~line 1421: introduced `array_init` function
  pointer that is `Value::zero` when init_zero is true,
  `Value::new` otherwise. Applied at both the named-element
  path (push_elem_named, line 1442) and the unnamed
  large-array path (line 1539). All array elements now zero
  at construction when INIT_ZERO=1.

  let array_init = if init_zero { Value::zero } else { Value::new };
  // ...
  sig_table.push(array_init(w));            // small-array path
  signal_table.extend(std::iter::repeat(array_init(w)).take(count));
                                            // large-array path

Verification:
  With fix + INIT_ZERO=1, the X-cascade is gone. CPU retires
  cleanly past the previous failure point. Confirmed by VCD
  showing first X transition no longer happens at t=42255.

Bug 2: Parallel-dispatch stall at sim 1,000,195 (workload-specific)
================================================================

Symptom:
  After fixing bug 1, c910 cmark hit a different failure: the
  testbench's "no instructions retired in 50000 cycles" watchdog
  fires at sim 1,000,195. CPU executed ~12 billion instructions
  but stopped retiring around sim ~500k. Same iters=200040
  signature as the prior settle-prefetch dead-end memo.

  c910 hello passed with parallel mode (printed "Hello Friend!");
  c910 cmark stalled. Workload-specific.

Root cause:
  The sequential bytecode interpreter (exec_insns) and the
  parallel-thread variant (exec_insns_isolated) implement the
  same Insn set, but with different state ownership:

   exec_insns:           uses `self.nba_fast` and
                          `self.nba_fast_index` (shared).
                          NbaAssignArray updates BOTH on each
                          push, so subsequent partial-range or
                          re-write Insns find existing entries
                          via the shared index.

   exec_insns_isolated:  uses thread-local `nba_out: Vec<NbaFast>`.
                          No shared index — each thread is blind
                          to other threads' writes during the
                          parallel scope.

  Classification at simulator.rs:4484 already excluded blocks
  containing partial-range / bit-dyn NBA Insns (NbaAssignRange,
  NbaAssignBitDyn, NbaAssignRangeDyn) from parallel mode for
  exactly this reason — those Insns specifically merge with
  prior entries via nba_fast_index.

  But `NbaAssignArray` (full unpacked-array element write) was
  STILL allowed in parallel-eligible blocks, even though it
  ALSO updates nba_fast_index in the sequential path. When two
  parallel-eligible blocks in different threads both write to
  the same array element, the chunk-by-range flatten ends up
  with both NbaFast entries in `nba_fast`, and the LAST one
  wins at apply time. That's last-writes-wins — same as
  sequential. So why does it break?

  The actual breakage is more subtle: when a SEQUENTIAL-only
  block (in Phase 2b) does NbaAssignRange to a signal that was
  also written by a parallel-eligible NbaAssignArray block in
  Phase 2a, the sequential block looks up nba_fast_index for
  the existing entry to merge with. After the parallel flatten,
  nba_fast_index is populated, so the lookup works. But the
  merge corrupts: the sequential block reads the FULL value
  from the parallel block's NbaFast entry (which contains all
  the bits of the array element from the parallel block's
  signal_table snapshot read), then mutates a sub-range. The
  problem: in the parallel path, multiple parallel blocks may
  have each pushed their own NbaFast for the same array
  element (cross-chunk). The "last one in nba_fast_index" is
  the one chunk-by-range happened to flatten last. Then the
  sequential block's range-merge corrupts THAT entry, but the
  earlier entries still apply (last-wins) and revert the
  merge. Cmark hits this on PRF / cache-line / ROB writes;
  hello doesn't.

Fix:
  simulator.rs:4498: add NbaAssignArray to the list of Insn
  types that disqualify a block from parallel-eligibility.
  Those blocks now run sequentially in Phase 2b (via
  exec_insns + self.nba_fast_index), where the cross-block
  coordination semantics work correctly. Other parallel-eligible
  blocks (NbaAssign for regular signals, all ALU comb, etc.)
  still parallelize.

  Same code-shape fix as the existing exclusions for the other
  partial / range / bit-dyn variants — fits the established
  pattern, minimally invasive.

Verification:
  With both fixes (INIT_ZERO array-init extension AND
  NbaAssignArray excluded from parallel):
   - c910 cmark with parallel mode + INIT_ZERO=1:
     * Sim 0 -> 1,100,000 (hit max-time, NOT a stall)
     * 51,680 sustained retire events, last at sim 1,099,935
     * 35,764,253,956 instructions executed
     * Zero TEST FAILED markers, zero errors
     * [PROF] par_dispatch partition=0 legacy=110003 — confirms
       parallel dispatch was actively used (110k parallel
       dispatches, no fallback to sequential)
   - c910 hello with parallel mode + INIT_ZERO=1:
     * Still passes, prints "Hello Friend!" correctly
     * Verified no regression from the array-write exclusion

  Pre-fix: TEST FAILED at sim 1,000,195 (deterministic, with
  iters=200040 watchdog signature). Post-fix: clean run to
  max-time. To actually reach TEST PASSED requires
  --max-time >= 2-3M sim units (Verilator golden ran to
  sim ~2,033,350) so cmark can finish computing the
  0x444333222 success magic.

Other diagnostic additions (left in place for future debugging)
================================================================

XEZIM_PARALLEL_SERIALIZE=1 (simulator.rs ~line 8889):
  When set, runs exec_insns_isolated sequentially in the main
  thread instead of dispatching to a thread pool. Useful for
  bisecting whether a future parallel-mode regression is in
  the function body's logic vs in the multi-thread scheduling
  / memory ordering. Default off; no perf impact when not set.
